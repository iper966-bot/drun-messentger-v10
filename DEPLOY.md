# Деплой на Render

Два способа: быстрый через Blueprint (`render.yaml`, уже лежит в проекте)
и обычный вручную. Если не уверены — используйте способ 1.

---

## Способ 1: Blueprint (автоматически, 1 файл — вся настройка)

### Шаг 1. Залейте проект в GitHub

Если ещё не в git:

```bash
cd drun-messenger
git init
git add .
git commit -m "Первый коммит"
```

Создайте пустой репозиторий на [github.com/new](https://github.com/new)
(приватный или публичный — не важно), затем:

```bash
git remote add origin https://github.com/ВАШ_ЛОГИН/НАЗВАНИЕ_РЕПО.git
git branch -M main
git push -u origin main
```

### Шаг 2. Создайте Blueprint на Render

1. Зайдите на [dashboard.render.com](https://dashboard.render.com)
   (зарегистрируйтесь, если ещё нет аккаунта — можно через GitHub).
2. Нажмите **New +** → **Blueprint**.
3. Подключите ваш GitHub-аккаунт, если ещё не подключён, и выберите
   репозиторий с проектом.
4. Render найдёт файл `render.yaml` в корне и покажет, какой сервис
   создаст (`drun-messenger`, план Free, диск 1 ГБ).
5. Нажмите **Apply**.

### Шаг 3. Задайте секреты

Blueprint создаст сервис, но два значения нужно ввести вручную —
Render специально не хранит их в `render.yaml`, чтобы они не попали
в git:

1. Откройте созданный сервис → вкладка **Environment**.
2. Найдите `OWNER_SIGNUP_CODE` — впишите свой код (минимум 8 символов,
   придумайте сами, это не пароль, а код для регистрации аккаунта
   "создателя").
3. `JWT_SECRET` Render уже сгенерировал сам случайным образом
   (`generateValue: true`) — трогать не нужно.
4. `ALLOWED_ORIGINS` можно оставить пустым, если фронтенд и бэкенд на
   одном домене (обычный случай для этого проекта).
5. Сохраните — сервис перезапустится с новыми переменными.

### Шаг 4. Готово

Через пару минут сборки сайт будет доступен по адресу вида
`https://drun-messenger-XXXX.onrender.com` (Render покажет точный
адрес на странице сервиса).

## Способ 2: вручную (без Blueprint)

Если Blueprint не подошёл (например, на бесплатном плане ограничение
на диски), настройте сервис вручную:

1. **New +** → **Web Service** → подключите репозиторий.
2. **Runtime**: Node
3. **Build Command**: `npm install`
4. **Start Command**: `npm start`
5. **Instance Type**: Free
6. Раздел **Environment Variables** — добавьте:
   - `NODE_ENV` = `production`
   - `JWT_SECRET` = длинная случайная строка (32+ символов). Сгенерировать
     локально: `node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"`
   - `OWNER_SIGNUP_CODE` = свой код доступа (8+ символов)
7. Раздел **Disks** (если доступен на вашем плане) — добавьте диск:
   - Name: `drun-data`
   - Mount Path: `/var/data`
   - Size: 1 GB
   Затем добавьте ещё две переменные окружения:
   - `DB_PATH` = `/var/data/nova.db`
   - `UPLOAD_DIR` = `/var/data/uploads`
8. Нажмите **Create Web Service**.

### Если диск недоступен (бесплатный план Render его не всегда даёт)

Без Persistent Disk файл базы `nova.db` и папка `uploads/` будут
**стираться при каждом передеплое** (это ограничение Render, не
проекта). Для тестового/демо-сайта это не критично; для постоянной
работы либо подключите платный план с диском, либо используйте внешнюю
БД — это уже выходит за рамки этого проекта "из коробки".

## Проверка после деплоя

1. Откройте адрес сайта — должна открыться страница входа/регистрации.
2. Зарегистрируйте обычный аккаунт — проверьте, что вход и переписка
   работают.
3. Зарегистрируйте аккаунт "создателя" через отдельную вкладку в форме
   регистрации, используя код, который вы задали в `OWNER_SIGNUP_CODE`.
4. Откройте DevTools (F12) → вкладку **Network** → обновите страницу —
   убедитесь, что запросы к `/api/...` не содержат секретов (токен
   только в заголовке `Authorization`, который и так должен быть на
   клиенте — это нормально для входа в систему).

## Частые проблемы

- **"Application failed to respond" / 502** — проверьте логи сервиса
  (вкладка **Logs**). Чаще всего это отсутствующий `JWT_SECRET` или
  `OWNER_SIGNUP_CODE` при `NODE_ENV=production` — сервер специально
  отказывается стартовать без них (см. `src/config.js`).
- **Node.js версия** — проект требует Node ≥22.5 (использует встроенный
  `node:sqlite`). Render обычно ставит последнюю LTS автоматически;
  если нет — задайте `NODE_VERSION=22.12.0` (или новее) в Environment.
- **Файлы/сообщения пропадают после деплоя** — значит `DB_PATH` и
  `UPLOAD_DIR` не указывают на смонтированный Persistent Disk. См.
  раздел про диск выше.
